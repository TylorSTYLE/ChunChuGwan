# 절차 C — 릴리즈 PR 생성 (develop → main)
공통 전제·금지·중단 조건은 SKILL.md를 따른다. 기본적으로 직접 병합하지 않으며 병합은 사용자가 수행한다.

**병합 방식 (불변)** — 릴리즈 PR은 **merge commit으로 병합한다. squash 금지.** squash 병합 시 main과 develop의 히스토리가 발산해 다음 릴리즈부터 이미 반영된 변경이 diff에 재등장하고 충돌이 발생한다.

**위임 원칙** — 병합 및 병합 후 태그 부여는 원칙적으로 사용자가 수행한다. 단, 사용자가 해당 작업을 명시적으로 위임한 경우 대행할 수 있다. 대행 시에도 (a) 확인 게이트(5·7단계)는 그대로 적용하고, (b) 병합은 merge commit으로 수행하며, (c) 태그 부여를 대행할 경우 병합된 main HEAD 커밋에 `v[버전]` 형식(예: v1.2.0)으로 정확히 부여하고 원격 push까지 완료하며, (d) 대행 여부·범위(병합만 / 병합+태그)를 최종 요약에 명시하며, (e) 병합 직전 gh pr checks <PR> --watch 초록 확인 + 병합·태그 push 직후 발화하는 워크플로(release.yml·docker-image.yml 등)를 gh run watch --exit-status로 성공까지 확인한다 — 실패 시 즉시 보고. 위임이 없거나 위임 범위가 불명확하면 대행하지 않고 사용자에게 넘긴다.

**태그 대행 제약** — 태그 부여 대행은 병합 대행을 함께 위임받은 경우로 한정한다. 병합을 사용자가 직접 수행하는 경우 태그만 단독 대행하지 않는다(대행 주체가 병합 완료 시점을 확정할 수 없어 잘못된 커밋에 태그가 붙을 위험이 있음). 따라서 위임 가능한 조합은 "병합만" 또는 "병합+태그" 둘 중 하나이며, "태그만" 위임은 불가하다. 사용자가 태그만 위임하려 하면 대행하지 않고, 병합까지 함께 위임할지 확인한다.

## 태깅 정책
릴리즈 PR이 main에 병합된 직후 반드시 해당 커밋에 `v[버전]` 태그(예: v1.2.0)를 달아야 한다. 태그가 없으면 다음 릴리즈 시 범위 파악이 불가능하므로 필수다. (태그 부여는 병합 후 사용자가 수행하며, 명시적 위임 시 대행 가능.)

## 수행 절차
1. **현재 상태 확인** — 작업 디렉터리가 dirty면 즉시 중단하고 보고한다.
2. **원격 동기화** — 삭제된 원격 브랜치 참조를 포함해 모든 원격 상태를 최신화한다.
3. **릴리즈 범위 파악** — 릴리즈 범위는 **develop에는 있고 main에는 아직 없는 커밋**이다. `git log origin/main..origin/develop` 기준으로 제시:
   - 직전 릴리즈 태그 확인(없으면 사용자에게 기준 커밋 확인)
   - main 대비 develop에 추가된 커밋 수·목록
   - 변경 파일 수·목록
   - 커밋 메시지의 Conventional Commits 프리픽스(`feat:`/`fix:`/`chore:`/`hotfix:`) 기반 유형 분류(squash 병합으로 PR 제목이 커밋 메시지에 보존됨)
   - **버전 자동 산출(SemVer)** — 직전 태그 이후 커밋 프리픽스 기준: BREAKING CHANGE 또는 타입 무관 `!` 표기(예: `feat!:`·`fix!:`) 포함=major / `feat:` 존재=minor / 그 외=patch. 등급을 결정한 커밋(산출 근거)과 함께 제안 버전을 제시한다. **직전 태그 이후 커밋이 0개면 릴리즈할 내용이 없다 — 즉시 중단하고 보고한다.**
4. **버전 파일 범핑 + 릴리스 노트 준비** — 3단계 산출 버전과 `pyproject.toml`의 `version`이 일치하는지, 그리고 `chunchugwan/web/release_notes.json`에 이 버전의 노트가 이미 반영돼 있는지 확인한다. 둘 중 하나라도 해당하면 `chore/prep-v[버전]` 브랜치를 생성해 develop 대상 준비 PR을 자동 작성한다(절차 A·B 준수 — PR 생성까지 자동, 병합은 위임 원칙에 따름):
   - 버전 불일치 시: `pyproject.toml`의 `version`을 산출 버전으로 갱신하고 `uv lock`으로 `uv.lock`을 동기화한다.
   - 릴리스 노트 미반영 시: `gh api repos/{owner}/{repo}/releases/generate-notes -f tag_name=v[버전] -f target_commitish=<develop HEAD SHA>`로 이번 버전 노트를 미리 받아 `python3 scripts/gen_release_notes.py [버전]`으로 변환해 `chunchugwan/web/release_notes.json`에 반영한다(실패해도 릴리스를 막지 않는다 — 이 버전은 인앱 업데이트 안내 모달만 비게 됨).

   준비 PR이 develop에 병합된 뒤 릴리즈를 재개한다 — 재개는 새 세션이어도 무방하며 반드시 1단계부터 다시 수행한다. (태그와 매니페스트 버전의 불일치, 그리고 인앱 업데이트 안내 모달 누락 방지)
5. **버전 확인 (확인 게이트)** — 3단계에서 자동 산출된 버전을 산출 근거와 함께 제시하고 사용자에게 확정받는다. 사용자가 다른 버전을 지정하면 그 값을 우선한다. 병합·태그 대행을 위임받은 경우 대행 범위도 함께 제시해 확인한다.
6. **PR 설명 작성** — 3단계의 프리픽스 분류를 기반으로 다음 형식으로 자동 작성:

## 릴리즈 [버전]
### 새 기능
- feat: 커밋 요약
### 버그 수정
- fix: 커밋 요약
### 유지보수
- chore: 커밋 요약 (CI/빌드/러너/캐싱 등, 해당 시)
### 긴급 수정
- hotfix: 커밋 요약 (해당 시)
### 변경 파일
- 변경 파일 수 및 주요 파일 목록

7. **PR 생성 승인 요청 (확인 게이트)** — 작성된 설명을 제시하고 진행 여부를 확인한다. 거부/무응답 시 즉시 중단.
8. **PR 생성** — develop → main 방향. 제목 `release: v[버전]`. 기본은 직접 병합 금지이며 병합은 사용자가 수행하되, **명시적으로 위임받은 경우 대행 가능**(merge commit 적용, 5·7단계 확인 게이트 통과 필수).
9. **결과 보고** — PR URL·요약 출력.
   - 병합을 대행하지 않는 경우: **merge commit으로 병합해야 함(squash 금지)**과, 병합 후 `v[버전]` 태그를 main HEAD에 달아야 함을 사용자에게 안내한다(태그 단독 대행 불가).
   - 병합+태그를 대행하는 경우: PR 병합(merge commit) 완료를 대행 주체가 직접 확인한 뒤 main HEAD 커밋에 `v[버전]` 태그를 부여하고 원격에 push한 후, 부여된 태그명·대상 커밋을 보고한다.
   - 병합만 대행하는 경우: 병합(merge commit)까지 대행하고, 태그는 사용자가 부여하도록 main HEAD에 `v[버전]` 태그가 필요함을 안내한다.
