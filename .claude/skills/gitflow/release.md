# 절차 C — 릴리즈 PR 생성 (develop → main)

공통 전제·금지·중단 조건은 SKILL.md를 따른다.

## 태깅 정책
릴리즈 PR이 main에 병합된 직후 반드시 해당 커밋에 `v[버전]` 태그(예: v1.2.0)를 달아야 한다. 태그가 없으면 다음 릴리즈 시 범위 파악이 불가능하므로 필수다. (태그 부여는 병합 후 사용자가 수행)

## 수행 절차
1. **현재 상태 확인** — 작업 디렉터리가 dirty면 즉시 중단하고 보고한다.
2. **원격 동기화** — 삭제된 원격 브랜치 참조를 포함해 모든 원격 상태를 최신화한다.
3. **릴리즈 범위 파악** — 직전 릴리즈 태그 이후 main에 추가된 모든 커밋 기준으로 제시:
   - 직전 릴리즈 태그 확인(없으면 사용자에게 기준 커밋 확인)
   - 직전 태그 이후 추가 커밋 수·목록
   - 변경 파일 수·목록
   - 브랜치 유형 분류(feature/*, bugfix/*, chore/*, hotfix/*)
4. **버전 확인 (확인 게이트)** — 릴리즈 버전을 사용자에게 확인한다.
5. **PR 설명 작성** — 다음 형식으로 자동 작성:
```
## 릴리즈 [버전]

### 새 기능
- feature/* 브랜치에서 병합된 항목 요약

### 버그 수정
- bugfix/* 브랜치에서 병합된 항목 요약

### 유지보수
- chore/* 브랜치에서 병합된 항목 요약 (CI/빌드/러너/캐싱 등, 해당 시)

### 긴급 수정
- hotfix/* 브랜치에서 병합된 항목 요약 (해당 시)

### 변경 파일
- 변경 파일 수 및 주요 파일 목록
```
6. **PR 생성 승인 요청 (확인 게이트)** — 작성된 설명을 제시하고 진행 여부를 확인한다. 거부/무응답 시 즉시 중단.
7. **PR 생성** — develop → main 방향. 제목 `release: [버전]`.
8. **결과 보고** — PR URL·요약 출력. 병합 후 `v[버전]` 태그를 main HEAD에 달아야 함을 사용자에게 안내한다.

## 진행 보고
각 단계 완료 후: ✅ [N단계 완료] — [실행한 내용 한 줄 요약]

전체 완료 후 최종 요약:
```
## 릴리즈 PR 생성 완료 요약
- 릴리즈 버전: [버전]
- PR 경로: develop → main
- 포함된 커밋: N개 / 변경 파일: N개
- PR URL: [URL]
- 병합 후: v[버전] 태그 부여 (사용자)
```
