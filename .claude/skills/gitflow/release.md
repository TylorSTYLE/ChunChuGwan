# 절차 C — 릴리즈 PR 생성 (develop → main)
공통 전제·금지·중단 조건은 SKILL.md를 따른다. 기본적으로 직접 병합하지 않으며 병합은 사용자가 수행한다.

**위임 원칙** — 병합 및 병합 후 태그 부여는 원칙적으로 사용자가 수행한다. 단, 사용자가 해당 작업을 명시적으로 위임한 경우 대행할 수 있다. 대행 시에도 (a) 확인 게이트(4·6단계)는 그대로 적용하고, (b) 병합은 GitFlow의 squash merge 전략을 따르며, (c) 태그 부여를 대행할 경우 병합된 main HEAD 커밋에 `v[버전]` 형식(예: v1.2.0)으로 정확히 부여하고 원격 push까지 완료하며, (d) 대행 여부·범위(병합만 / 병합+태그)를 최종 요약에 명시한다. 위임이 없거나 위임 범위가 불명확하면 대행하지 않고 사용자에게 넘긴다.

**태그 대행 제약** — 태그 부여 대행은 병합 대행을 함께 위임받은 경우로 한정한다. 병합을 사용자가 직접 수행하는 경우 태그만 단독 대행하지 않는다(대행 주체가 병합 완료 시점을 확정할 수 없어 잘못된 커밋에 태그가 붙을 위험이 있음). 따라서 위임 가능한 조합은 "병합만" 또는 "병합+태그" 둘 중 하나이며, "태그만" 위임은 불가하다. 사용자가 태그만 위임하려 하면 대행하지 않고, 병합까지 함께 위임할지 확인한다.

## 태깅 정책
릴리즈 PR이 main에 병합된 직후 반드시 해당 커밋에 `v[버전]` 태그(예: v1.2.0)를 달아야 한다. 태그가 없으면 다음 릴리즈 시 범위 파악이 불가능하므로 필수다. (태그 부여는 병합 후 사용자가 수행하며, 명시적 위임 시 대행 가능.)

## 수행 절차
1. **현재 상태 확인** — 작업 디렉터리가 dirty면 즉시 중단하고 보고한다.
2. **원격 동기화** — 삭제된 원격 브랜치 참조를 포함해 모든 원격 상태를 최신화한다.
3. **릴리즈 범위 파악** — 직전 릴리즈 태그 이후 main에 추가된 모든 커밋 기준으로 제시:
   - 직전 릴리즈 태그 확인(없으면 사용자에게 기준 커밋 확인)
   - 직전 태그 이후 추가 커밋 수·목록
   - 변경 파일 수·목록
   - 브랜치 유형 분류(feature/*, bugfix/*, chore/*, hotfix/*)
4. **버전 확인 (확인 게이트)** — 릴리즈 버전을 사용자에게 확인한다. 병합·태그 대행을 위임받은 경우 대행 범위도 함께 제시해 확인한다.
5. **PR 설명 작성** — 다음 형식으로 자동 작성:
```
## 릴리즈 [버전]
### 새 기능
- feature/* 브랜치에서 병합된 항목 요약
### 버그 수정
- bugfix/* 브랜치에서 병합된 항목 요약
### 유지보수
- chore/* 브랜치에서 병합된 항목 요약 (CI/빌드/러너/캐싱 등, 해당 시)
### 긴급 수정
- hotfix/* 브랜치에서 병합된 항목 요약 (해당 시)
### 변경 파일
- 변경 파일 수 및 주요 파일 목록
```
6. **PR 생성 승인 요청 (확인 게이트)** — 작성된 설명을 제시하고 진행 여부를 확인한다. 거부/무응답 시 즉시 중단.
7. **PR 생성** — develop → main 방향. 제목 `release: [버전]`. 기본은 직접 병합 금지이며 병합은 사용자가 수행하되, **명시적으로 위임받은 경우 대행 가능**(squash merge 전략 적용, 4·6단계 확인 게이트 통과 필수).
8. **결과 보고** — PR URL·요약 출력.
   - 병합을 대행하지 않는 경우: 병합 후 `v[버전]` 태그를 main HEAD에 달아야 함을 사용자에게 안내한다(태그 단독 대행 불가).
   - 병합+태그를 대행하는 경우: PR 병합(squash) 완료를 대행 주체가 직접 확인한 뒤 main HEAD 커밋에 `v[버전]` 태그를 부여하고 원격에 push한 후, 부여된 태그명·대상 커밋을 보고한다.
   - 병합만 대행하는 경우: 병합(squash)까지 대행하고, 태그는 사용자가 부여하도록 main HEAD에 `v[버전]` 태그가 필요함을 안내한다.

## 진행 보고
각 단계 완료 후: ✅ [N단계 완료] — [실행한 내용 한 줄 요약]
전체 완료 후 최종 요약:
```
## 릴리즈 PR 생성 완료 요약
- 릴리즈 버전: [버전]
- PR 경로: develop → main
- 포함된 커밋: N개 / 변경 파일: N개
- PR URL: [URL]
- 병합: 사용자 수행 예정 / 대행 완료(위임, squash)
- 태그 부여: 사용자 수행 예정 / 대행 완료(위임, v[버전] → [커밋])
  ※ 태그 대행은 병합 대행을 함께 위임한 경우에만 가능(태그 단독 대행 불가)
```
