# 절차 A — 작업 브랜치 생성·재개

공통 전제·금지·중단 조건은 SKILL.md를 따른다. 여기서는 브랜치 절차만 다룬다.

## 브랜치 유형 판단 기준
| 유형 | 사용 시점 | 베이스 | 병합 대상 |
|------|----------|--------|----------|
| feature/[작업명] | 새 기능 개발 | develop | develop (사용자 병합) |
| bugfix/[작업명] | develop 단계 버그 수정 | develop | develop (사용자 병합) |
| chore/[작업명] | 기능·버그와 무관한 유지보수 (CI/CD·빌드·러너·캐싱·의존성 정리·문서 등) | develop | develop (사용자 병합) |
| hotfix/[작업명] | main 운영 긴급 수정 | main | main PR + develop PR (사용자 병합) |

## 신규 생성
1. 작업 내용을 파악해 유형과 브랜치명 후보를 사용자에게 제시한다. **(확인 게이트)**
2. 사용자 확인 시 베이스 브랜치를 원격 최신 상태로 갱신한 뒤 브랜치를 생성한다.
3. 원격에 push하고 추적 설정을 완료한다.

## 기존 브랜치 재개
리베이스 기준 — feature/*·bugfix/*·chore/*: develop / hotfix/*: main.
다음을 파악해 제시하고 리베이스 진행 여부를 확인한다 **(확인 게이트)**:
- 작업 브랜치가 기준 브랜치 대비 몇 커밋 뒤처져 있는지
- 기준 브랜치에 새로 추가된 커밋 목록과 각 커밋의 변경 파일
- 작업 브랜치와 기준 브랜치 간 잠재적 충돌 가능 파일

동의 시 기준 브랜치를 최신화한 뒤 작업 브랜치에 리베이스한다. 충돌 시 즉시 중단하고 충돌 파일 목록을 보고한다.
