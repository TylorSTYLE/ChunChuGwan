# 텍스트 검색 (Full-text search)

아카이브된 모든 스냅샷의 **본문 텍스트**와 **첨부 문서(PDF·워드·한글 등)
본문**을 전문 검색한다. 외부 검색엔진 없이 기존 `archive/index.db`(SQLite)
안의 FTS5 인덱스만 쓴다 — 별도 서비스·설정이 필요 없다.

## 무엇을 검색하나

- **페이지 본문** — 스냅샷의 `content.md`(추출+정규화된 텍스트). 타임스탬프·
  광고 등 노이즈가 제거된, diff/해시와 같은 기준의 깨끗한 본문이다.
- **첨부 문서 본문** — 스냅샷에 함께 저장된 문서 파일에서 추출한 텍스트.
  지원 형식: **PDF, DOCX, PPTX, XLSX, ODT/ODP/ODS, HWPX, EPUB**.
  구형 바이너리(`.doc`/`.ppt`/`.xls`/`.hwp` 등)와 압축/독점 형식
  (`.zip`/`.pages`/`.key`/`.numbers`)은 본문 추출 대상이 아니라 파일명·
  메타데이터만 검색된다.
- **제목과 URL** — `meta.json` 의 title 과 페이지 URL 도 함께 색인된다.

## 한국어 검색 특성 (꼭 읽을 것)

토크나이저는 **trigram**(3글자 슬라이딩 부분문자열)이다. 한국어는 조사가
붙어 띄어쓰기 단어가 검색어와 잘 안 맞으므로, 단어분절 대신 부분문자열
검색이 적합하다.

- **3글자 이상**(`헌법재판`, `검색엔진`)이 가장 정확하다 — 조사·어미와 무관하게
  부분문자열로 찾는다.
- **1~2글자**(`법`, `세금`) 쿼리는 trigram 인덱스가 못 잡으므로 **부분일치
  (LIKE) 폴백**으로 처리한다. 결과는 나오지만 인덱스를 안 타므로 스냅샷이
  아주 많으면 느릴 수 있다 — 가능하면 도메인 필터와 함께 쓰자.
- **여러 단어**는 AND 로 묶인다 — `검색엔진 구현` 은 두 단어를 모두 포함하는
  스냅샷을 찾는다.
- trigram 은 **형태소 검색이 아니다** — `먹었다`/`먹는다` 처럼 활용형은 의미로
  묶지 못하고, 우연히 3글자가 겹치는 오탐이 섞일 수 있다.

## CLI

```bash
uv run wccg search <검색어>                 # 전문 검색 (여러 단어 = AND)
uv run wccg search 헌법재판 --domain example.com   # 도메인 한정
uv run wccg search 키워드 --latest          # URL 당 최신 스냅샷 1건만
uv run wccg search 키워드 --limit 50        # 결과 수 (기본 20)
uv run wccg search status                   # 인덱스 상태 / 미색인 스냅샷 수
uv run wccg search reindex                  # 미색인 스냅샷 백필
uv run wccg search reindex --all            # 전체 재색인 (인덱스 비우고 다시)
uv run wccg search verify                   # 정합성 점검 (과소 색인·orphan 감지)
uv run wccg search verify --repair          # 발견한 불일치 교정
```

`status` 는 정상적인 미색인(pending) 수만 보여준다. `verify` 는 한 걸음 더
나아가 **플래그와 실제 FTS 행의 불일치** — *과소 색인*(색인됨으로 표시됐지만
FTS 행이 없음)과 *orphan*(스냅샷이 없는 FTS 행) — 을 잡는다. 이 부류는
pending 카운트만으로는 보이지 않는다. `--repair` 는 orphan 을 지우고 과소
색인 스냅샷을 다시 색인한다 (전체를 비우는 `reindex --all` 보다 가볍다).

## 대시보드

**현황 화면 상단의 검색 박스**에서 진입한다(`/search`, viewer 이상 —
헤더 메뉴에는 따로 두지 않는다). 검색어·도메인 필터·"URL당 최신만" 토글을
제공하고, 결과마다 매치 부분을 강조한 스니펫과 스냅샷 뷰어·타임라인 링크를
보여준다.

> 전문 검색은 사실상 모든 아카이브 본문을 열람하는 강한 권한이라 **보기 전용
> (viewer) 이상**만 쓸 수 있다. 인증이 꺼진 loopback 환경에서는 누구나 가능.

**시스템 메뉴**(`/system`, 관리자 전용)의 **검색 인덱스** 카드는 인덱스
상태(색인됨·미색인·과소 색인·orphan)를 보여주고, **전체 다시 색인** 버튼으로
인덱스를 비우고 전체 스냅샷을 다시 색인한다 (`wccg search reindex --all` 과
동일 — 과소 색인·orphan·stale 을 한 번에 교정). 동기 실행이라 스냅샷이 많으면
시간이 걸릴 수 있다.

## 색인은 언제 만들어지나

- **새 스냅샷**은 저장 시점에 자동으로 색인된다 (아카이빙 파이프라인이 같은
  트랜잭션에서 처리). 별도 작업이 필요 없다.
- **기존(이 기능 도입 전) 스냅샷**, **가져오기(import)로 들어온 스냅샷**,
  드물게 **색인에 실패한 스냅샷**은 `search_indexed=0` 으로 남아 검색에 잡히지
  않는다. **`wccg search reindex`** 를 한 번 돌리면 일괄 색인된다
  (`wccg compact` 와 같은 멱등 백필 패턴).
- 스냅샷을 삭제하면 인덱스에서도 함께 제거된다. `compact` 는 본문(content.md)을
  바꾸지 않으므로 페이지 본문은 재색인이 필요 없다 — 다만 `compact` 가 구형
  스냅샷의 `files/` 문서를 문서 CAS 로 이전하면(첨부 문서가 새로 인식됨) 그
  스냅샷을 자동으로 "다시 색인 필요"(`search_indexed=0`)로 표시한다. 이후
  `search reindex`(또는 시스템 메뉴의 전체 다시 색인)가 그 문서 본문까지
  색인한다. `compact` 뒤에 `search status` 에 미색인 수가 보이면 이 때문이다.

## 정합성

검색 인덱스(`search_indexed` 플래그 + `snapshot_fts` 행)와 아카이브 데이터의
정합성은 다음으로 유지된다:

- **삭제**는 `db.delete_snapshot`/`delete_page` 한 곳을 거치며 FTS 행도 함께
  지운다 (단일/페이지/사이트 삭제·overwrite 가져오기 모두 커버). → orphan 미발생.
- **본문 불변** — `content.md` 는 생성 시 1회 기록 후 바뀌지 않는다(원칙 2).
  → 페이지 본문 색인은 stale 되지 않는다.
- **첨부 문서 변동**은 `compact` 의 self-heal(위)로 다시 색인 대상이 된다.

정상 경로 밖(외부 DB 조작·파일 손상·백필 실패)으로 어긋날 수 있는 두 부류는
`wccg search verify` 로 감지하고 `--repair` 또는 **전체 다시 색인**으로 교정한다:
*과소 색인*(플래그=1·FTS 행 없음)과 *orphan*(FTS 행·스냅샷 없음).

## 동작하지 않을 때

- `wccg search status` 가 **"비활성"** 이라고 나오면 이 환경의 SQLite 빌드에
  **FTS5 가 없는** 것이다 (대부분의 배포판·도커 이미지는 FTS5 포함). 이 경우
  검색 기능만 비활성화되고 기존 아카이빙은 영향받지 않는다.
- 검색 인덱스는 재생성 가능한 파생 데이터다 — `wccg export` 에는 포함되지
  않으며(가져온 뒤 `reindex` 로 만든다), 전체 백업(`wccg backup`)에는 DB 의
  일부로 그대로 포함된다.
